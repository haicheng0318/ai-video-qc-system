# AI短视频质检评估系统 V1.0 - Codex 项目开发指令 ## 一、项目定位 本项目名称为：AI短视频质检评估系统 V1.0。 本项目是内容中台内部使用的 AI 短视频质检与有效产出评定系统，用于判断短视频编导交付的视频是否为有效视频。 本系统不是普通的视频上传系统，也不是单一 AI 打分系统。 系统需要实现从视频提交、AI 内容评估、主管初审、运营/投放数据补充、AI 数据复盘、规则引擎判断、最终评定到案例沉淀的完整业务闭环。 --- ## 二、核心业务流程 系统核心流程如下： 编导提交视频 → Gemini 视频内容质量评估 → 主管初审 → 运营/投放补充结果数据 → GPT 数据复盘分析 → 后端规则引擎判断 → GPT 最终有效等级评定 → 负责人确认 → 沉淀有效视频、优秀案例、反面案例。 任何开发任务都必须服务于以上业务闭环。 不要把系统简化成“上传视频 + AI 打分”。 --- ## 三、系统最终要输出的三个等级 系统必须分别输出并保存三个等级： 1. 内容质量等级 - 由 Gemini 视频内容质量评估产生； - 用于判断视频本身做得好不好。 2. 数据表现等级 - 由 GPT 数据复盘智能体产生； - 用于判断视频上线/投放后的运营或投放数据表现好不好。 3. 最终有效等级 - 由后端规则引擎 + GPT 最终评定智能体 + 负责人确认共同产生； - 用于判断视频是否计入有效视频、是否进入案例库、是否可用于绩效参考。 这三个等级必须独立存储，不能混为一个字段。 --- ## 四、智能体分工 本项目必须遵守以下智能体分工： ### 1. Gemini Gemini 只负责视频内容质量评估。 Gemini 负责判断： - 前 3 秒是否有吸引力； - 产品是否清晰露出； - 卖点是否表达清楚； - 画面质感是否达标； - 节奏是否拖沓； - 字幕/口播是否清晰； - 是否符合平台和视频用途； - 是否存在合规风险； - 是否建议发布或投放。 Gemini 不负责： - 运营/投放数据复盘； - 最终有效等级评定； - 绩效判断。 --- ### 2. GPT GPT 负责两个环节： 第一，运营/投放数据复盘。 GPT 需要根据运营/投放补充的数据，判断： - 数据表现是否达标； - 数据样本是否充分； - CTR、CVR、ROI、进房率、完播率等指标是否异常； - 问题更可能来自内容、投放、人群、价格、商品详情页、直播间承接、活动机制还是样本不足； - 视频是否具备继续测试、优化或复用价值。 第二，最终有效等级建议。 GPT 需要基于： - Gemini 内容评估结果； - 主管初审结果； - 运营/投放数据； - GPT 数据复盘结果； - 后端规则引擎结果； 生成最终等级建议。 GPT 不负责直接替代负责人确认。 --- ### 3. 规则引擎 规则引擎必须是后端代码，不是 Prompt。 规则引擎负责最终评定前的硬边界判断。 必须实现以下规则： - data_sufficiency = insufficient → pending_data； - 内容 S/A + 数据 S/A → excellent_effective_candidate； - 内容 S/A + 数据 B → effective_candidate； - 内容 B + 数据 S/A → potential_effective_candidate； - 内容 B + 数据 B → basic_effective_candidate； - 内容 S/A + 数据 C/D → content_good_result_poor； - 内容 C/D + 数据 S/A → abnormal_need_confirmation； - 内容 C/D + 数据 C/D → invalid_candidate。 规则引擎结果必须保存到 rule_engine_results 表，并传给 GPT 最终评定智能体。 --- ### 4. Claude Claude 不参与本系统的日常逐条视频评估。 Claude 的定位是： - 架构审查； - 代码质量复核； - 业务逻辑补强； - 权限安全检查； - 状态流转检查； - AI 接入逻辑检查。 不要把 Claude 接入为日常视频评估模型。 --- ## 五、V1.0 必须实现的功能 V1.0 必须实现： 1. 用户登录与角色权限； 2. 视频上传； 3. 视频列表； 4. 视频详情； 5. Gemini 内容质量评估； 6. 内容质量等级； 7. 主管初审； 8. 返修版本关联； 9. 运营/投放结果数据补充； 10. GPT 数据复盘； 11. 数据表现等级； 12. 后端规则引擎； 13. GPT 最终评定； 14. 负责人最终确认； 15. 数据看板； 16. 优秀案例库； 17. 反面案例库； 18. 操作日志。 --- ## 六、V1.0 暂不实现的功能 V1.0 暂不实现： 1. 自动抓取抖音数据； 2. 自动抓取千川数据； 3. 自动抓取天猫/京东数据； 4. 自动计算绩效工资； 5. 自动剪辑视频； 6. 自动生成视频； 7. 企业微信/飞书深度审批流； 8. 多公司复杂组织权限； 9. 自动投放建议； 10. 自动关联订单系统。 如果用户后续要求上述功能，先作为 V1.1 或后续版本建议，不要擅自加入 V1.0。 --- ## 七、推荐技术栈 优先采用以下技术栈： - 前端：Next.js + React； - 后端：FastAPI 或 NestJS； - 数据库：PostgreSQL； - 鉴权：JWT； - 文件存储：本地测试 + 后续对象存储扩展； - 视频内容模型：Gemini API； - 数据分析模型：OpenAI GPT API； - 最终评定模型：OpenAI GPT API； - 部署：Docker + 云服务器； - 异步任务：Celery 或 BullMQ。 如果实际技术选型发生变化，必须说明原因。 --- ## 八、必须保留的核心数据表 数据库必须包含以下核心表： 1. users； 2. videos； 3. ai_content_reviews； 4. content_review_scores； 5. supervisor_reviews； 6. video_result_metrics； 7. ai_result_reviews； 8. rule_engine_results； 9. final_video_evaluations； 10. operation_logs； 11. ai_model_configs； 12. platform_benchmarks。 这些表不能随意省略。 --- ## 九、关键数据表说明 ### 1. ai_content_reviews 用于保存 Gemini 视频内容质量评估结果。 必须保存： - model_provider； - model_name； - content_summary； - total_score； - content_grade； - is_publishable_recommendation； - main_problems； - revision_suggestions； - compliance_risks； - usable_scenarios； - raw_response； - status； - error_message。 --- ### 2. ai_result_reviews 用于保存 GPT 运营/投放数据复盘结果。 必须保存： - model_provider； - model_name； - data_score； - data_grade； - data_sufficiency； - is_business_effective_recommendation； - result_summary； - performance_problems； - attribution_analysis； - optimization_suggestions； - raw_response； - status； - error_message。 --- ### 3. rule_engine_results 用于保存后端规则引擎判断结果。 必须保存： - content_grade； - data_grade； - data_sufficiency； - rule_code； - rule_result； - rule_reason； - recommended_boundary。 --- ### 4. final_video_evaluations 用于保存最终有效等级评定结果。 必须保存： - content_review_id； - result_review_id； - rule_engine_result_id； - model_provider； - model_name； - content_grade； - data_grade； - final_grade； - final_status； - is_effective_final； - is_excellent_case； - is_negative_case； - can_be_used_for_performance； - final_attribution； - final_suggestion； - raw_response； - confirmed_by； - confirmed_at； - manual_adjust_reason。 --- ### 5. operation_logs 用于保存关键操作日志。 必须记录： - 上传视频； - Gemini 内容评估完成； - 主管初审； - 运营/投放数据补充； - GPT 数据复盘完成； - 规则引擎执行； - GPT 最终评定完成； - 负责人确认； - 人工调整最终等级； - 标记优秀案例； - 标记反面案例。 --- ## 十、用户角色与权限 系统必须支持以下角色： 1. 管理员； 2. 内容负责人； 3. 编导主管； 4. 编导； 5. 运营； 6. 投放。 权限原则： - 编导只能查看自己提交的视频； - 编导不能修改 AI 结果； - 编导不能修改运营/投放数据； - 编导不能确认最终等级； - 主管可以查看团队视频并进行初审； - 运营可以补充运营数据； - 投放可以补充投放数据； - 运营/投放不能修改最终等级； - 内容负责人可以确认最终评定； - 管理员可以管理全部数据； - 权限必须在后端校验，不能只靠前端隐藏按钮。 --- ## 十一、视频状态流转 视频必须按以下状态流转： submitted → ai_content_reviewing → pending_supervisor_review → approved_for_publish / revision_required / invalid_content → pending_result_data → ai_result_reviewing → pending_rule_engine → pending_final_evaluation → pending_final_confirmation → final_effective / final_low_effective / final_invalid / pending_data。 异常状态必须包含： - ai_content_failed； - ai_result_failed； - final_evaluation_failed。 状态流转要求： - Gemini 内容评估失败时，状态变为 ai_content_failed； - GPT 数据复盘失败时，状态变为 ai_result_failed； - GPT 最终评定失败时，状态变为 final_evaluation_failed； - 主管判定返修时，返修版本必须创建新记录，并通过 parent_video_id 关联原视频； - 主管判定无效时，不进入后续数据补充流程； - 数据不足时，最终状态必须是 pending_data； - 负责人确认后，最终状态才能进入 final_effective / final_low_effective / final_invalid。 --- ## 十二、视频类型 系统至少支持以下视频类型： 1. 商品卡视频； 2. 千川投放视频； 3. 直播间引流视频； 4. 自然流视频； 5. 品牌种草视频； 6. 其他。 不同视频类型必须支持不同运营/投放数据字段。 --- ## 十三、运营/投放归因类型 GPT 数据复盘必须支持以下归因类型： 1. content：内容问题； 2. delivery：投放设置问题； 3. audience：人群问题； 4. price：价格问题； 5. product_page：商品详情页承接问题； 6. live_room：直播间承接问题； 7. activity：活动机制问题； 8. sample_size：数据样本不足； 9. external：外部异常因素。 注意： - ROI 低不能直接归因为编导问题； - 数据不足不能判定为无效； - 内容好但数据差，要进入归因排查； - 内容差但数据好，要进入人工复核。 --- ## 十四、AI 输出要求 Gemini 和 GPT 输出必须是结构化 JSON。 必须做到： - 使用 JSON Schema 校验； - 解析失败时进入失败状态； - 支持重新触发； - 保存 raw_response； - 不覆盖原始 AI 结果。 --- ## 十五、API Key 与安全要求 - Gemini API Key 只能存储在后端环境变量； - GPT API Key 只能存储在后端环境变量； - 前端不得出现任何模型 API Key； - 日志中不得打印完整 API Key； - 环境变量文件不得提交到 GitHub； - 视频文件访问需要权限控制； - 权限必须由后端接口校验。 --- ## 十六、开发阶段要求 不要一次性开发完整系统。 必须按阶段推进： ### 第一阶段 只做基础系统： 1. 初始化前后端项目； 2. 配置数据库； 3. 创建基础数据表； 4. 创建默认管理员账号； 5. 实现登录和 JWT 鉴权； 6. 实现角色权限； 7. 实现视频上传； 8. 实现视频列表； 9. 实现视频详情； 10. 预留 Gemini、GPT、规则引擎相关模块目录和数据表，但暂时不要正式接入 Gemini / GPT API。 ### 第二阶段 接入 Gemini 内容质量评估。 ### 第三阶段 开发主管初审和返修流程。 ### 第四阶段 开发运营/投放数据补充。 ### 第五阶段 接入 GPT 数据复盘。 ### 第六阶段 开发后端规则引擎。 ### 第七阶段 接入 GPT 最终评定。 ### 第八阶段 开发负责人确认、看板和案例库。 ### 第九阶段 配合 Claude 做架构审查和代码质量复核。 --- ## 十七、开发禁止事项 不要做以下事情： 1. 不要一次性开发完整系统； 2. 不要擅自修改 PRD 的业务流程； 3. 不要把 Gemini 用于数据复盘或最终评定； 4. 不要把 GPT 用于视频文件内容理解； 5. 不要把规则引擎只写在 Prompt 里； 6. 不要让 AI 最终结果直接等于业务最终确认； 7. 不要覆盖 AI 原始返回结果； 8. 不要省略 operation_logs； 9. 不要跳过后端权限校验； 10. 不要把 API Key 写在前端； 11. 不要把数据不足判定为 D； 12. 不要把 ROI 低直接归因为编导问题； 13. 不要删除或合并核心数据表； 14. 不要在没有说明的情况下改变技术栈。 --- ## 十八、每次交付格式 每完成一个阶段，请按以下格式输出： 1. 已完成内容； 2. 修改文件清单； 3. 新增接口清单； 4. 新增数据表或字段； 5. 启动方式； 6. 测试方式； 7. 当前已知问题； 8. 与 PRD 的差异； 9. 下一步建议。 --- ## 十九、第一阶段完成后的验收标准 第一阶段完成后，必须满足： 1. 前端可以启动； 2. 后端可以启动； 3. 数据库可以连接； 4. 默认管理员可以登录； 5. 用户角色权限生效； 6. 视频可以上传； 7. 视频可以播放； 8. 视频列表可以查看； 9. 视频详情可以查看； 10. 核心数据表已创建； 11. Gemini / GPT / 规则引擎相关模块已经预留； 12. API Key 未暴露； 13. README 包含启动和测试说明。 --- ## 二十、最终目标 本系统最终目标不是生成一个 AI 分数，而是建立一套可追踪、可复盘、可管理的短视频有效产出判断体系。 请始终围绕以下原则开发： Gemini 看视频， GPT 看数据和最终建议， 规则引擎控制硬边界， 负责人确认最终结果， Claude 负责审查和纠偏。
